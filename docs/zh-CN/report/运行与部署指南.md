# 运行与部署指南

## 适用范围

- 单容器 Docker 部署
- 双容器兼容部署
- Linux 二进制部署
- 反向代理场景

## 默认推荐

- 默认推荐：单容器 `ghcr.io/chengliang4810/codexmanager:latest`
- 兼容方案：双容器 `codexmanager-service` + `codexmanager-web`

## 默认端口

- 对外入口：`48761`
- 内部 service 端口：`48760`

在单容器模式下，通常只需要暴露 `48761`；`48760` 仅用于容器内部进程通信。

## Docker Compose 部署（推荐，单容器）

直接在项目目录创建 `docker-compose.yml`，内容如下：

```yaml
services:
  codexmanager:
    image: ghcr.io/chengliang4810/codexmanager:latest
    restart: unless-stopped
    environment:
      CODEXMANAGER_WEB_ACCESS_PASSWORD: admin123 # 首次访问密码，部署后可在页面修改
    volumes:
      - ./data:/data # 持久化数据库和 RPC token 到当前目录 ./data
    ports:
      - "48761:48761" # Web 管理页与对外 /v1 入口
```

启动命令：

```bash
mkdir -p ./data
docker compose up -d
```

说明：

- 单容器镜像会以 `codexmanager-web` 作为主进程，并自动拉起同镜像内的 `codexmanager-service`
- 如果需要页面内在线升级，请保留 `restart: unless-stopped`

访问地址：

- `http://localhost:48761/`
- `http://localhost:48761/v1/...`

## Docker 直接部署（推荐，单容器）

```bash
mkdir -p ./data

docker run -d \
  --name codexmanager \
  --restart unless-stopped \
  -p 48761:48761 \
  -v "$(pwd)/data:/data" \
  -e CODEXMANAGER_WEB_ACCESS_PASSWORD=admin123 \
  ghcr.io/chengliang4810/codexmanager:latest
```

补充说明：

- 单容器 `codexmanager` 镜像内部已经内置下面这些默认值，不需要用户手动重复配置：
  - `CODEXMANAGER_SERVICE_ADDR=0.0.0.0:48760`
  - `CODEXMANAGER_WEB_ADDR=0.0.0.0:48761`
  - `CODEXMANAGER_DB_PATH=/data/codexmanager.db`
  - `CODEXMANAGER_RPC_TOKEN_FILE=/data/codexmanager.rpc-token`
  - `CODEXMANAGER_WEB_NO_OPEN=1`
- 对外通常只需要开放 `48761`
- 如果使用 `./data:/data` 目录映射，镜像启动时会自动修正 `/data` 目录权限，再切回应用用户运行，通常不需要额外手工 `chown ./data`

## 双容器兼容部署（可选）

如果你明确需要拆分 `service` / `web` 两个容器，可以继续使用下面的兼容模式：

```yaml
services:
  codexmanager-service:
    image: ghcr.io/chengliang4810/codexmanager-service:latest
    restart: unless-stopped
    environment:
      CODEXMANAGER_WEB_ACCESS_PASSWORD: admin123
    volumes:
      - ./data:/data

  codexmanager-web:
    image: ghcr.io/chengliang4810/codexmanager-web:latest
    restart: unless-stopped
    depends_on:
      codexmanager-service:
        condition: service_healthy
    volumes:
      - ./data:/data
    ports:
      - "48761:48761"
```

说明：

- `codexmanager-web` 镜像默认使用 `CODEXMANAGER_SERVICE_ADDR=codexmanager-service:48760`
- 双容器模式下页面内在线升级不可用，建议通过拉新镜像再重启容器完成更新

## Linux 二进制部署

```bash
# 解压 Release 附件后
chmod +x codexmanager-service codexmanager-web codexmanager-start

# 推荐：直接启动 web，由它自动拉起同目录下的 service
CODEXMANAGER_WEB_ACCESS_PASSWORD=admin123 \
./codexmanager-web
```

如果你需要手动拆分二进制，也可以继续按下面的方式分别启动：

```bash
./codexmanager-service

CODEXMANAGER_WEB_NO_SPAWN_SERVICE=1 \
CODEXMANAGER_SERVICE_ADDR=127.0.0.1:48760 \
./codexmanager-web
```

也可以使用：

```bash
./codexmanager-start
```

## 升级方式

### 单容器模式

单容器 `codexmanager` 镜像支持页面内在线升级：

1. 确保容器使用了 `restart: unless-stopped`
2. 在页面里检查更新
3. 点击“立即更新”
4. 程序会下载 Release 二进制、校验 `checksums.txt`、替换容器内程序并主动退出
5. 依赖 Docker restart policy 自动重新拉起

### 双容器模式

双容器 `codexmanager-service` + `codexmanager-web` 当前仍建议手动拉取新镜像，再重启容器。

## 反向代理要求

如果要把 `codexmanager-web` 放在 Nginx / Caddy 后面，至少要转发：

- `/api/runtime`
- `/api/rpc`
- `/v1/*`
- `/health`
- `/metrics`
- 静态页面路由

只托管静态文件会导致管理页面无法完成 RPC 调用。

## 常见问题

### 页面能打开，但管理操作失败

优先检查：

1. 单容器或双容器里的 `codexmanager-service` 是否健康
2. `codexmanager-web` 是否能访问到内部 service
3. 反向代理是否放通 `/api/runtime`、`/api/rpc`、`/v1/*`

### Docker 里 web 连不上 service

优先检查：

1. 如果是单容器模式，是否误设了 `CODEXMANAGER_WEB_NO_SPAWN_SERVICE=1`
2. 如果是双容器模式，两个容器是否在同一 Docker network
3. `CODEXMANAGER_SERVICE_ADDR` 是否写成了正确的容器服务名
